运行核心剖析：把一次聊天编译成可验收的执行图

UltraGameStudio 不把任务直接丢给模型，而是先编译成一张有依赖关系的执行图，再交给一个宿主无关的引擎调度。本文拆开 app/src/runtime/：IRGraph 怎么被执行、节点如何重试与背压、Schema 与共识如何把不可控的文本收敛成可验收的结果。

$ executeWorkflowDag(graph, ctx, callbacks)
# 拓扑入度 0 的节点先就绪
ready: [classify]
dispatchNode -> runParallel | runPipeline | runConsensus
runNodeWithRetry: attempt 0..N, backoff 1.5s*n

引擎只认图、上下文、回调和网关。窗口、stdout、Tauri 命令都在宿主层，不进入任务语义。
01 / MOTIVATION

为什么不把复杂任务塞进一次模型调用

最直接的做法是把整个任务写成一个长 prompt，让模型一口气给出答案。这在单步问答里够用，但游戏开发里的真实任务往往是多步的：先判断属于哪类问题，再并行收集证据，改代码，跑验收，最后汇总。把这些步骤压进一次调用，意味着失败时你只能看到一团文本，既无法定位是哪一步出错，也无法只重跑失败的部分。

UltraGameStudio 的运行核心选择把任务显式建成一张图。每个节点是一次可观测的执行单元，节点之间用依赖关系连接。调度器按拓扑顺序推进，能并行的并行，需要前置结果的等待，失败的节点按策略重试或让整图失败。这样做的代价是接口更明确、概念更多；收益是任务可恢复、可审计，而且同一套图能在桌面端和 CLI 上跑出完全一致的行为。

这张图就是 IRGraph。它既是 /studio 动态生成执行计划的产物，也是普通聊天会话的底层载体。引擎入口 executeWorkflowDag 不关心图是动态生成还是预置的，它只负责把图跑完并产出终态。

本文范围 只讲运行核心的执行语义：图、调度、网关、验收、策略。不涉及 UI 布局、资产生成模式和具体的 provider 配置。

02 / IR GRAPH

IRGraph 是引擎运行其上的唯一底物

任务在执行前先被表达成节点和依赖。节点有类型，决定它如何被分派；依赖决定它何时就绪。引擎不读懂自然语言意图，它只读这张结构。

node

一个执行单元，携带 prompt、可选的模型与网关覆盖、可选的 Schema 名称。普通 agent 节点直接调一次模型；复合节点展开成并行分支、流水线阶段或共识投票。

edge

表达依赖。buildDependencyGraph 把边折算成每个节点的入度；入度归零的节点进入就绪集合，由 getRunnableNodes 取出。

context

RunContext 注入网关、个性化指令、超时与速度启发式。它是引擎够到模型的唯一通道，不直接持有 HTTP 客户端或子进程。

meta.schemaDefs

图级别的 Schema 字典。节点用名字（如 VERDICT）引用，引擎据此对该节点输出做结构校验和有界重试。

关键约束写在源码的 CONTRACT: 注释里：runtime/ 对宿主是纯的——不 import React / Zustand / Tauri，不碰 window / document / localStorage。所有副作用走注入的回调，所有模型能力走注入的网关。
03 / SCHEDULING

调度：拓扑推进、并行收敛、有界重试

executeWorkflowDag 是主循环。它反复取出就绪节点，分派执行，记录终态，再根据完成情况解锁后继节点，直到没有可跑的节点为止。

就绪 入度为 0 的节点进入就绪集合。
分派 dispatchNode 按类型走 parallel / pipeline / consensus 或单次调用。
重试 可重试失败按 1.5s×n、上限 15s 退避，最多 N 次。
记录 写入 NodeRunResult：状态、输出、耗时、失败码、重试次数。
解锁 后继节点入度递减，归零即就绪，循环继续。
runParallel

把多个分支并发跑，受 runWithConcurrency 的并发上限约束。这是背压所在：分支再多，同时在飞的请求数也被限住，避免一次性打满上游通道。

runPipeline

顺序阶段，前一阶段的输出作为后一阶段的上下文。适合"先草稿、后精修"这类需要逐步收敛的链路。

runConsensus

多个投票者各自产出，再由 resolveConsensus 按策略裁决。对抗式验收在这里落地：让一组模型互相检查，而不是相信单次输出。

runNodeWithRetry

统一的单节点执行加重试。失败先被 parseRunFailure 分类（timeout / idle_timeout / spawn / backend …），只有可重试类型才进退避循环。

04 / GATEWAY

网关把"调用模型"和"宿主怎么调"彻底分开

节点要调一次模型，走的是 RunGateway。它抽象掉三件事：直接 HTTP completion、CLI 子进程 spawn、网关路由解析。桌面端把它实现成 Tauri 的 aiEditViaCli spawn 接缝加 Zustand 流式渲染；Node CLI 把它实现成 child_process 加 stdout 和 readline。两个宿主消费同一份 runtime/ 代码，所以同一张图跑出来的 DAG 顺序、重试、终态完全一致。

模型不是一次性吐完答案就结束。runAgentWithInteraction 在 invokeAgent 外面包了一层有界的"节点可以反问用户"循环：每一轮通过 callbacks.beginStream 流式输出，如果模型按交互协议发出一个询问块，引擎渲染成交互控件、阻塞等待 promptInteraction 的答复，把答复拼回上下文再跑下一轮。回合数有上限（MAX_INTERACTION_ROUNDS），避免模型把用户拖进无限问答。

这层设计让交互成为引擎能力而不是 UI 补丁。桌面端把交互渲染成可点按钮，CLI 把它渲染成命令行提问，但"什么时候该问、问几轮、答复怎么回填"由核心统一决定。

失败分类 RunFailureCode 把失败收敛成有限集合：timeout、idle_timeout、interrupted、exit、spawn、backend、wait、unknown。重试策略只对可重试类型生效，认证类 4xx 这种不可重试的会直接上报，不浪费退避时间。
05 / GUARDS

Schema 和共识：把不可控的文本收敛成可验收的结果

模型输出是自由文本，但下游往往需要结构化、可信的结果。引擎用两道独立的闸门处理这件事。

Schema 约束

节点引用一个 Schema 名，引擎在 prompt 末尾追加结构说明，用 extractJson 从输出里抽出 JSON，再用 validateAgainstSchema 校验。不合格时 schemaRetryFeedback 把具体问题回喂给模型做有界重试，而不是直接接受脏数据。

共识否决

共识节点让多个投票者产出，再由策略裁决。对抗式策略允许验证者否决——只要有否决，结果就不通过，迫使链路重新生成而不是放行一个看似合理但错误的答案。

这两道闸门的共同点是：它们不假设模型一定对。Schema 处理"格式不对"，共识处理"内容可能错"。配合 DAG 的重试，整图在单个模型不稳定时仍然能逼近一个可交付的结果，而不是把第一份输出当成最终答案。

所有这些过程都落到 .ugs-run/<run-id>/：事件流、任务账本、每个节点选了哪个模型、最终 verdict。失败时能精确回答是哪一步、哪个通道、哪个验收条件出了问题——这正是把任务建成图而不是塞进一次调用所换来的可观测性。

06 / STRATEGIES

六种动态策略：planner 按任务形状选编排

/studio 不套固定模板。planner 读任务后从六种策略里选一种或组合，生成对应形状的执行图。下面是每种策略解决的问题。

策略 图的形状 适用任务
classify-and-act 先分类，再按类别走不同分支。 意图不确定的请求，先判断属于哪类问题再决定怎么处理。
fan-out-and-synthesize 并行铺开多个子任务，再汇聚成一个综合结果。 可拆成独立子问题的任务，并发收集后由一个收敛节点合并。
adversarial-verification 生成与验证分离，验证者可否决。 正确性要求高的改动，用对抗检查替代"相信单次输出"。
generate-and-filter 先多产候选，再按条件过滤。 需要从一批方案里挑出合格项，而不是一次生成就用。
tournament 候选两两/分组比较，逐轮淘汰。 有多个看似都行的方案，需要择优而非简单合并。
loop-until-done 循环执行直到满足验收条件。 结果需要迭代逼近的任务，每轮检查是否达标再决定是否继续。

这些策略不是互斥的标签，而是图的构造模式。当并行宽度超过阈值时，引擎会自动插入一个收敛节点把扇出结果归并，避免下游被一堆零散输出淹没。策略选择、模型分配、验收节点最终都体现成 IRGraph 的结构，再交给同一个 executeWorkflowDag 跑完。

核心判断：复杂任务的稳定性来自结构，不来自更长的 prompt。把编排显式化，引擎就能在并行、重试、验收和恢复上提供保证，而这些保证在桌面端和 CLI 上是同一套。
